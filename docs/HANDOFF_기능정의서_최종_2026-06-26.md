### 2.4 nginx (FE 배포)

```nginx
server {
  listen 8080;
  location /api/ {
    proxy_pass https://innocean-audience-api-291757702623.asia-northeast3.run.app/api/;
    proxy_http_version 1.1;
    proxy_read_timeout 180s;
    proxy_send_timeout 180s;
  }
  location / {
    root /usr/share/nginx/html;
    try_files $uri $uri/ /index.html;
    add_header Cache-Control "no-store, no-cache, must-revalidate" always;
  }
  location /health { return 200 "ok\n"; }
}
```

- `no-store` 강제로 캐시 회피 — 매 배포마다 강력 새로고침으로 즉시 반영
- proxy_read_timeout 180s — Gemini 합성 장시간 요청 보호

### 2.5 데이터 영속 (SQLite + GCS)

- 파일: `backend/src/lib/persona-store.js`, `persona-gcs.js`, `routes/library.js`
- 환경변수:
  - `PERSONA_DB_PATH` (default `./data/audience-personas.db`)
  - `PERSONA_GCS_BUCKET` (필수, 미설정 시 no-op)
  - `PERSONA_GCS_OBJECT` (default `audience-personas.db`)
- 테이블:
  - `campaign_briefs(id, ...)`
  - `campaign_personas(persona_id, brief_id, country, attrs JSON, ...)`
  - `campaign_insights((brief_id, scope, country, tab), data JSON)`
  - `campaign_generation_state((brief_id, country), status/progress)`
  - `library_items(id, owner, name, country, filters, tags, favorited, recent, saved, created_at, updated_at)` ← **2026-06-26 SQLite 영속화 완료**
- **PRAGMA journal_mode = WAL** (동시성 + 충돌 회피)
- **부트 플로우**:
  1. `downloadDbFromGcs()` (5s timeout, 실패 시 silent skip — 빈 DB로 시작)
  2. `Database.open()` (better-sqlite3, WAL)
  3. `_setDbAccessor(getDb)` (gcs 모듈이 동일 DB 핸들 사용)
- **쓰기 플로우**:
  1. 배치 쓰기 → `scheduleDbUpload()` 호출
  2. 30초 debounce 후 GCS 업로드 (fire-and-forget, 실패는 warn-only)
- **종료 플로우** (SIGTERM): `installShutdownHook()` → `forceDbUpload("SIGTERM")` 동기 실행
- **다중 인스턴스 주의**: min/max-instances=1 권장. 다중 인스턴스 시 last-write-wins (§11)

## §5. AI 생성 파이프라인

### 5.1 단계 (campaign-personas /generate)

```
POST /api/personas/brief         → brief 저장 (countries/targets/regions/sizePerCountry)
POST /api/personas/generate      → setState(running) + runGeneration() fire-and-forget
                                    ↓
                       buildCohort(country, 100)        ← cohort-builder.js (통계 기반 속성)
                                    ↓
                       synthesizeNarratives(batch=10, concurrency=2)
                                    ↓ (per batch)
                       gemini.generateJSON({
                         model:"gemini-2.5-flash",
                         thinkingBudget: 0,
                         timeoutMs: 90_000,
                         maxRetries: 3,
                         responseSchema: NARRATIVE_SCHEMA_ITEM[]
                       })
                                    ↓ 실패/누락
                       per-persona 단발 재호출 → 그래도 실패 → fallbackNarrative(p)
                                    ↓
                       _normalizeScoreObject / _normalizeEnum / _sanitizeGender
                                    ↓
                       appendPersonas() (SQLite WAL)
                                    ↓
                       (모든 국가 완료) buildInsightPayload(all) → setInsights() → setState(done)
                                    ↓
                       scheduleDbUpload() (30s debounce → GCS)
```

### 5.2 Gemini 어댑터 핵심 (`adapters/gemini.js`)

| Export | 모델 default | 옵션 |
|---|---|---|
| `isGeminiAvailable()` | — | `GCP_PROJECT` 존재로 판단 |
| `generateText({prompt, system, model, temperature, maxTokens, thinkingBudget})` | gemini-2.5-flash | thinkingBudget=0 default (잘림 방지) |
| `generateJSON({prompt, system, model, responseSchema, thinkingBudget=0, timeoutMs=90000, maxRetries=3})` | gemini-2.5-flash | AbortController + race + exponential backoff + 파싱 실패 retry |
| `chat({messages, system, model, temperature=0.7, maxTokens=2048, thinkingBudget=0})` | gemini-2.5-flash | 인터뷰 라우트 사용 |
| `searchAndSummarize({query, model, maxTokens})` | gemini-2.5-flash (us-central1) | Google Search grounding 3 variant 폴백 (googleSearch/google_search/googleSearchRetrieval) + 최종 grounding 없이 generate |

**CEO 절대 정책**:
- `thinkingBudget=0` 기본 — thinking 토큰이 `maxOutputTokens`을 잠식해 답변 잘리는 버그 방지 (2026-06)
- timeout 90s + retry 3회 + backoff — 무한 hang 방지 (2026-06-26 Critical fix §K1)

### 5.3 부정 키워드 가드 (safety.js)

- `sanitizeText`, `isUserAskingAboutNegative`, `SAFETY_PROMPT_GUIDE`
- 차단 키워드: 논란/우울/범죄/안티팬/혐오/자살/마약 등
- system prompt에 자동 주입 + 응답 후 한번 더 sanitize

### 5.4 신조어·외래어 표기 (fix 39)

```
형식: 한국어 표기(원어, 발음): 의미 한 줄 설명
예) 평타족(平躺族, 핑탕쭈): 노력하지 않고 평평하게 누워 산다는 중국 청년 문화
```

## §6. 솔루션 사용 프로세스 — 경우의 수 전부

> **공통 원칙**: 6국(KR/JP/CN/TW/TH/PH) = persona-pool 모드 가능, 그 외 = public-data 모드(BASELINE 또는 거시지표만).

### 6.1 (A) 프리셋 사용 — 미리 만든 풀 즉시 표시

1. 사이드바 "⭐ 타겟 프리셋" → 프리셋 모달
2. `fashion-2034-global` (3국 KR/JP/CN, age 20-34, 의류/K-패션) 선택
3. 빌더 자동 채움 + briefId 자동 연결 (기존 풀 재사용)
4. 분석 시작 → 즉시 **persona-pool** 모드(파란 배지) — 사전 생성된 100명/국 풀 표시
5. 6탭 + 인터뷰 + 미디어 즉시 가용

### 6.2 (B) 빌더 직접 설정 — 분석 시작과 동시에 풀 생성

1. 사이드바 "📊 타겟 인사이트" → mode-insight overview 탭
2. 국가/6대 차원/관심사/세그먼트 다중 선택
3. "분석 시작" 클릭 → `POST /api/personas/brief` + `/generate` 자동 호출
4. **분석 직후 표시**: public-data(일반) 모드 — 즉시 BASELINE으로 6탭 채움
5. 풀 생성 진행 중 → `/api/personas/status` 폴링 (총 ~10-18분/국, GenerateBadge "분석 중")
6. 풀 완성 → 자동으로 persona-pool(파란 배지) 모드로 스왑 + 6탭 재집계
7. 인터뷰/미디어 탭 활성화 (페르소나 풀 SoT)

### 6.3 (C) 자연어 채팅 (ad-hoc brief)

1. mode-chat 홈 → 추천 칩 또는 textarea 자연어 질문 입력
2. `POST /api/ask` (Gemini 2.5 Flash structured + 룰베이스 fallback) → `{ countries, filters, intent }` 추출
3. briefId 없는 경우 → ad-hoc brief 자동 생성 + `/api/personas/generate` 즉시 트리거
4. `POST /api/insight` 호출 — 첫 질문은 **전체 인사이트 보고서**(headline+narrative+KPI+차트+페르소나샘플+sources+relatedInsights), 후속 질문은 **채팅 버블만**(headline+narrative)
5. 동일 세션 후속 질문 = **무조건 후속 처리** (CEO 2026-06-16 11:55), priorTurns 6턴 전달
6. "라이프스타일 표준 분석 →" Related 버튼 → `chatBoardJumpToTab('life')` → mode-insight 진입 + 빌더 자동 세팅 + `runAnalysis()`

### 6.4 (D) 국가 전환 (KR/JP/CN/TW/TH/PH)

1. 빌더 또는 채팅 칩에서 국가 변경
2. 백엔드: 국가별 독립 brief (이미 있으면 재사용, 없으면 신규 생성)
3. 프론트: 캐시 invalidate(`state.audienceCache[code]` 비우기) → 새로 fetch
4. **국가 swap race 가드** (2026-06-26 §K3): 진행 중 swap 요청은 토큰 비교로 stale 응답 폐기, 완료된 것은 보존
5. 풀 미존재 국가는 즉시 public-data 모드 → 백그라운드 생성 후 swap

### 6.5 (E) 탭 전환 — 단일 풀 SoT

1. 동일 brief 안에서 7탭 + 인터뷰 + 미디어 + 글로벌비교 모두 **같은 풀**에서 산출
2. `aggregateAll(personas)` → byTab 캐시 (insights 테이블) + on-the-fly fallback
3. 탭 전환은 캐시 hit으로 즉시 렌더

### 10.2 환경변수 (값 비밀, 키 이름만)

|---|---|---|

**⚠️ 시크릿은 Cloud Run secret 또는 IAM 자동 인증 사용. 코드/리포지토리/이 문서에 값 노출 절대 금지.**

### 10.3 배포 명령

#### 백엔드
```bash
gcloud run deploy innocean-audience-api \
  --source backend \
  --region=asia-northeast3 \
  --min-instances=1 --max-instances=1 \
  --memory=512Mi --cpu=1 \
  --no-cpu-throttling \
  --set-env-vars PERSONA_GCS_BUCKET=...,GCP_PROJECT=...,VERTEX_LOCATION=asia-northeast3
```

#### 프론트엔드 (nginx 정적)
```bash
gcloud run deploy innocean-audience \
  --source frontend \
  --region=asia-northeast3 \
  --min-instances=1 --max-instances=1 \
  --memory=256Mi --cpu=1
# ※ FE는 --no-cpu-throttling 사용 금지 (nginx 정적, 메모리 256Mi 충분)
```

**Docker `--no-cache` 빌드** — Cloud Run source 빌드는 자동 적용. 강력 새로고침 + nginx `Cache-Control: no-store` 조합으로 클라이언트 캐시 회피.

### 10.4 검증 절차

1. 코드 수정 → 따옴표 이스케이프 검증 (Python 스크립트 assert)
2. `git commit -m "..." && git push origin main`
3. Cloud Build 자동 트리거 → 배포 완료
4. **curl 키워드 검증**: 라이브 URL `curl -s ... | grep -E "키워드"` → ALL PASSED
5. 강력 새로고침(Cmd+Shift+R) 안내
6. CEO 보고: 결론 → 근거 → 다음 액션 (정중체, ✅/❌)

### 10.5 GCS 버킷 정책

- 1개 오브젝트(`audience-personas.db`) 덮어쓰기 모델
- min/max-instances=1 권장 (다중 인스턴스 시 last-write-wins)
